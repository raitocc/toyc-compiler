# ToyC 编译器性能优化设计方案

在 ToyC 编译器的性能评测中，运行时间得分占总评测得分的 25%，并以 `gcc -O2` 作为满分基准。目前我们的代码生成策略（Naive Code Generation）存在大量的内存读写冗余，执行效率较低。

为了在开启 `-opt` 参数时能够大幅缩短生成汇编的运行时间，我们计划在编译流程的各个阶段引入以下几项核心优化算法。

---

## 1. 寄存器分配 (Register Allocation) —— 性能提升的核心大杀器

**问题现状**：  
当前我们在生成的汇编中，每一个局部变量和临时变量都被死板地分配到了堆栈内存（栈帧偏移量 `-4(s0)`, `-8(s0)` 等）。这意味着即使是 `a + b` 这样简单的操作，也会产生 `lw`, `lw`, `add`, `sw` 四条指令，其中包含三条极其耗时的内存访问指令。

**优化方案**：  
实现一个基础的**线性扫描寄存器分配器 (Linear Scan Register Allocator)** 或者**贪心分配器**。
* RISC-V 提供了大量的空闲寄存器（如保存寄存器 `s1` ~ `s11`，以及临时寄存器 `t0` ~ `t6`）。
* 在 `-opt` 开启时，我们对 IR 指令遍历，将频繁存取、生命周期较短的临时变量和局部变量优先映射到物理寄存器中。
* 只有当活变量（Live variables）数量超过物理寄存器数量时，才触发“溢出 (Spill)”机制，将它们存入栈内存。
* **预期效果**：大幅消除冗余的 `lw` 和 `sw` 指令，让运算完全在 CPU 内部完成，极大概率能逼近 `gcc -O1/O2` 的效率！

## 2. 窥孔优化 (Peephole Optimization)

**问题现状**：  
在从 IR 线性生成汇编时，相邻的两条汇编指令常常会产生逻辑冗余。比如刚刚存入栈的值，立马在下一条指令中被读出：
```assembly
sw t2, -12(s0)
lw t0, -12(s0)
```

**优化方案**：  
在生成完汇编代码链表后，设立一个“滑动窗口（大小通常为 2~4 条指令）”对生成的汇编流进行后处理扫描：
* **消除冗余 Load/Store**：如果发现对同一个内存地址发生了先写后读（且寄存器不变或可以安全复用），直接砍掉后面的 `lw`，改为 `mv` 或直接服用寄存器。
* **分支跳转合并**：比如 `j label1`，而 `label1` 的下一条指令恰好就是 `j label2`，可以直接将其折叠为 `j label2`。
* **常量指令折叠**：比如连续的两条 `addi sp, sp, 16` 可以折叠成 `addi sp, sp, 32`。

## 3. 常量折叠与传播 (Constant Folding & Propagation)

**问题现状**：  
当前即使代码里写着 `int c = 3 + 5 * 2;`，编译器也会尽职尽责地生成对应的 `ADD` 和 `MUL` 指令，让计算机在运行时去算 `13`。对于全局变量我们在语义分析时做了求值，但局部变量和 IR 阶段还没有。

**优化方案**：  
* **常量折叠**：在生成 IR 或者遍历 AST 时，如果发现操作数全部为已知常数（如 `ConstValue`），直接在编译期算出结果，用一个常量节点代替复杂的表达式节点。
* **常量传播**：顺着数据流向下查找，如果确定变量 `a` 被赋值为了常量 `5`，且后续没有被修改，就把后面所有用到 `a` 的地方全部替换成直接使用字面量 `5`（RISC-V 中的 `li` 指令）。

## 4. 死代码消除 (Dead Code Elimination - DCE)

**问题现状**：  
在复杂的源程序或者因为前面的常量折叠等优化步骤产生后，往往会遗留大量在运行时永远不会执行、或其计算结果永远不会被使用的指令。

**优化方案**：  
* **不可达代码消除**：在 `return`、`break`、`continue` 语句之后，如果没有标签（Label）跳转作为入口的代码块，直接在 AST 或 IR 层级将其删去。
* **无用赋值消除**：遍历 IR，如果发现一个局部变量/临时变量被赋值了（`ASSIGN`），但它在之后的作用域生命周期中从未被读取过，且表达式没有副作用（比如函数调用），那么这条赋值指令和它的计算过程就可以安全地删掉。

---

## 实施路径建议

如果决定开始优化，推荐按以下顺序入手（按性价比排序）：
1. **寄存器分配**（直接消灭掉 80% 的内存读写，性价比最高，可以直接挑战满分！）
2. **窥孔优化**（实现难度低，只需在汇编生成后加一个滑动数组扫描即可）。
3. **常量折叠 / 死代码消除**（如果在前端做会比较简单）。
