# ToyC IR 优化收益分析

本文只讨论在 IR 阶段继续优化能够获得的收益，不包含寄存器分配、汇编窥孔优化和后端指令选择。当前 `-opt` 已经启用局部常量折叠、局部常量/复写传播、简单代数化简、粗粒度死代码删除，以及线性扫描寄存器分配。Benchmark 中仍然与 `gcc -O2` 有明显差距，主要原因不是寄存器分配一个点，而是大量可以在 IR 层直接蒸发的计算仍然被完整生成并执行。

## 1. 当前性能差距的形态

从 Benchmark 的 `GCC/Opt = GCC cyc / IR-Opt cyc` 可以看出，普通循环 mock 用例大多在 `0.6 ~ 0.8` 左右，说明寄存器分配和局部优化已经能吃到一部分收益。但递归、死循环、可常量求值函数调用类用例差距非常大：

| 用例 | 现象 | 主要差距来源 |
| --- | --- | --- |
| `perf_deadcode` | `GCC/Opt` 约 `0.01` | GCC 直接删除无副作用循环，ToyC 仍执行千万级 IR |
| `perf_hanoi` | `GCC/Opt` 约 `0.05` | 递归调用开销被指数级放大，GCC 后端和中端都更强 |
| `perf_fib` | `GCC/Opt` 约 `0.1` | 常量实参递归调用未被求值，函数调用与栈帧开销极大 |
| `perf_gcd` | `GCC/Opt` 约 `0.17` | 循环内重复调用 `gcd(48, 18)`，ToyC 没有常量调用折叠 |
| `mock_p07_loop` | `GCC/Opt` 约 `0.7` | 以全局变量作为循环边界，GCC 优势被削弱，ToyC 仍有全局 load 和循环内冗余 |

这说明后续 IR 优化的目标不应只是让已有 IR 运行更快，而应尽量让 IR 指令数、循环次数、函数调用次数在编译期直接消失。

## 2. GCC 在这些用例中做了什么

### 2.1 常量折叠与表达式简化

`perf_deadcode` 中有大量如下表达式：

```c
sum = sum + (100 * 200 / 50 + 400 - 300) * 0;
```

GCC 会把右侧完整折叠成 `sum + 0`，继续化简成 `sum`。如果赋值对后续没有实际影响，整条语句也可以删除。

当前 ToyC 已有基本块内常量折叠，但传播能力有限，无法稳定把跨多条临时变量的结果一路折叠到最终赋值，更无法因此删除循环。

### 2.2 死代码消除与无副作用循环删除

当循环体只修改局部变量，且这些修改不影响最终返回值，也没有函数调用或全局写入时，GCC 可以删除整个循环。`perf_deadcode` 的 `sum` 始终为 `0`，循环变量 `i` 只用于控制循环次数，循环本身没有副作用，因此可以直接变成：

```c
return 0;
```

当前 ToyC 的死代码删除只是统计“某个临时变量是否在函数内被用过”。这种方法过粗，无法处理“变量被用来计算自己但最终值不变”的情况。

### 2.3 纯函数常量求值

对于：

```c
return fib(35);
return hanoi(25);
sum = sum + gcd(48, 18);
```

如果函数体不读写全局状态，也没有 I/O 或未定义行为，那么常量实参调用可以在编译期解释执行。GCC 不一定总是把递归完全算完，但它的内联、常量传播、调用优化和后端优化会显著降低开销。

ToyC 语言没有数组、指针和 I/O，函数副作用来源很少，因此非常适合做保守的纯函数常量求值。

### 2.4 循环不变量外提

循环中只依赖常量或循环外值的表达式可以移到循环前。例如全局只读变量、重复出现的常量表达式、与循环变量无关的计算，都不应在每次迭代中重新计算。

当前 ToyC 缺少循环分析，也没有在 IR 中标记自然循环，因此这类优化还没有做。

### 2.5 控制流清理

当前 IR 生成会留下 `RET` 后的 `JMP`：

```text
RET t0
JMP -> if_end_2
```

这类指令虽然运行时不可达，但会干扰后续数据流分析，也可能增加寄存器分配压力。GCC 在早期 CFG pass 中会清掉这些基本块和边。

## 3. IR 阶段可以优先实现的优化

### 3.1 CFG 清理

目标：

1. 删除 `RET`、无条件 `JMP` 后面的不可达指令。
2. 从入口块开始遍历 CFG，删除不可达基本块。
3. 常量条件分支改写为确定跳转或空操作。
4. 删除跳向下一基本块的冗余 `JMP`。
5. 合并只有单一前驱、单一后继的简单基本块。

预期收益：

- 直接减少 IR 指令数。
- 降低后续活跃变量分析和寄存器分配的复杂度。
- 为更强 DCE、常量传播、循环分析提供干净 CFG。

实现风险：

- 低。只要 CFG 边构造正确，语义基本不会变。

### 3.2 全局只读变量常量传播

目标：

1. 扫描所有 `GlobalVar` 初始值。
2. 扫描函数 IR，记录被写入的全局变量。
3. 对从未被写入的全局变量，将读取替换成 `ConstValue(initValue)`。

示例：

```c
int LIMIT = 10000000;
while (i < LIMIT) { ... }
```

可以在 IR 中改成：

```text
t = SLT i, 10000000
```

预期收益：

- mock 循环类用例收益明显。
- 让常量折叠、分支折叠、循环分析有更多机会继续向下传播。

实现风险：

- 中低。ToyC 没有指针，判断全局变量是否被写入相对简单。
- 需要注意全局变量不是 C 语义上的 `const`，只读传播只在全程序内没有赋值时才安全。

### 3.3 真正的反向死代码消除

目标：

用活跃变量集合从基本块末尾向前扫描：

1. 如果指令定义了临时变量，且该变量不在 live 集合中，并且指令无副作用，则删除。
2. 如果保留指令，则从 live 中移除其定义，加入其使用。
3. 对基本块之间使用 `liveOut` 做定点迭代，处理跨块活跃变量。

相比当前粗粒度 DCE，这种方法可以删除更多链式临时变量，并为循环删除创造条件。

预期收益：

- 大幅减少代数化简后残留的无用计算。
- 对 `perf_deadcode` 这类用例是必要前置优化。
- 对寄存器分配也有间接收益，活跃区间会变短。

实现风险：

- 中。需要准确区分有副作用指令：`CALL`、写全局变量、控制流跳转、`RET` 必须保留。
- 局部变量和临时变量在 IR 中都是 `TempVar`，删除赋值时要保证不影响后续读取。

### 3.4 稀疏条件常量传播的简化版

目标：

在 CFG 上传播变量状态：

- `未知`
- `常量 c`
- `非常量`

当分支条件为常量时，只沿可达分支传播。这样可以同时完成常量传播和不可达分支删除。

示例：

```c
if (1) {
    return 3;
} else {
    return expensive();
}
```

优化后只保留 then 分支。

预期收益：

- 比基本块内传播强，能跨分支、跨循环前后传播。
- 可删除更多不可达基本块。

实现风险：

- 中高。完整 SCCP 实现较复杂，可以先做保守版本，只在无歧义时替换。

### 3.5 纯函数常量求值

目标：

在优化阶段对满足条件的函数调用做编译期解释：

1. 被调函数不写全局变量。
2. 被调函数只调用同样可解释的函数。
3. 调用实参全是常量。
4. 解释执行有步数上限，防止编译期爆炸。
5. 对 `(funcName, args...)` 做 memoization，专门优化递归。

可优化示例：

```c
fib(35)
hanoi(25)
gcd(48, 18)
ack(3, 6)
```

预期收益：

- 对递归性能用例收益极大。
- `perf_gcd` 中循环内的 `gcd(48, 18)` 可直接变成常量 `6`。
- 常量调用折叠后，后续 DCE 和循环删除可以继续工作。

实现风险：

- 中高。需要写一个小型 IR 解释器。
- 必须有 fuel 限制，例如最多解释若干百万步，超出就放弃优化而不是报错。
- 需要遵守 32 位 int 溢出语义，保持与 RISC-V/GCC 结果一致。

### 3.6 循环不变量外提

目标：

识别自然循环，把循环体内满足条件的纯计算移到循环前：

1. 指令无副作用。
2. 操作数来自常量、循环外定义，或已经被判定为循环不变量。
3. 目标临时变量不在循环外被错误使用。

预期收益：

- 对长循环 mock 用例有收益。
- 能减少重复 `li`、`la/lw`、算术计算。

实现风险：

- 中高。需要循环识别、preheader 构造和支配关系的简化处理。
- 可以先只处理 `while_cond` / `while_body` 这种由当前 IR 生成器固定产生的结构。

### 3.7 无副作用循环删除

目标：

在常量传播和 DCE 后，如果一个循环只修改循环内部变量或最终不被使用的局部变量，且没有函数调用、全局写、返回、break 逃逸副作用，则删除整个循环。

预期收益：

- `perf_deadcode` 可以从千万级循环降到接近常量返回。

实现风险：

- 高。循环删除一旦误判，正确性会直接出问题。
- 建议放在 CFG 清理、反向 DCE、常量传播之后，并且只做非常保守的模式匹配。

## 4. 推荐实施顺序

### 阶段一：清理 IR 骨架

1. CFG 清理。
2. 删除 `RET/JMP` 后不可达指令。
3. 删除不可达基本块。
4. 常量条件分支折叠。

这一阶段收益不一定最大，但能明显降低后续优化复杂度。

### 阶段二：增强常量与死代码能力

1. 全局只读变量常量传播。
2. 强化代数化简，例如 `x + 0`、`x - 0`、`x * 0`、`x / 1`、`x % 1`、比较常量折叠。
3. 反向活跃变量 DCE。

这一阶段能改善大多数 mock 用例，并减少寄存器压力。

### 阶段三：常量调用求值

实现带 fuel 和 memoization 的纯函数 IR 解释器。先支持：

1. 算术、比较、赋值。
2. `BEQZ`、`BNEZ`、`JMP`、`RET`。
3. 函数调用。
4. 局部变量环境。

优先验证 `gcd(48, 18)`、`hanoi(25)`、`fib(35)`、`ack(3, 6)`。这是最可能显著拉高 `GCC/Opt` 的 IR 优化。

### 阶段四：循环优化

1. 简化版循环不变量外提。
2. 极保守无副作用循环删除。

这部分收益高，但正确性风险也更高，适合在前面优化稳定后再做。

## 5. 与当前用例的对应关系

| 优化 | 主要命中的用例 | 预期效果 |
| --- | --- | --- |
| CFG 清理 | `fib`、`hanoi`、所有 if/while 用例 | 减少无用跳转，清理不可达块 |
| 全局只读常量传播 | `mock_p07_loop`、`mock_p08_basic_combined`、`mock_p12_const_expr_chain` | 循环边界和全局常量转为立即数 |
| 反向 DCE | `perf_deadcode`、代数化简类 mock | 删除链式无用临时计算 |
| 纯函数常量求值 | `perf_gcd`、`perf_fib`、`perf_hanoi`、`perf_ackermann` | 将常量实参函数调用折叠为常量 |
| 循环不变量外提 | 深层循环与矩阵类 mock | 减少循环内重复计算 |
| 无副作用循环删除 | `perf_deadcode` | 整段循环蒸发 |

## 6. 结论

继续优化 IR 的性价比很高。当前后端寄存器分配已经能让普通循环 mock 接近 GCC 的一半以上，但 `perf_deadcode`、`perf_fib`、`perf_hanoi` 这种用例暴露的是中端能力不足：能在编译期算掉或删掉的程序仍然被完整执行。

最推荐先做 CFG 清理、全局只读常量传播、反向 DCE，再做带 memoization 的纯函数常量求值。前两步是地基，第三步能减少普遍冗余，第四步最可能带来性能分数上的大跳跃。
