# ToyC 语义分析阶段需求细化文档

本文档记录了 ToyC 编译器在语义分析阶段（上下文相关检查）针对程序结构和函数的具体要求，并将其细化拆分为代码实现中的检查步骤。

---

## 1. 程序结构要求

### 原始文档要求

> 程序可以包含全局变量声明、全局常量声明和函数定义。程序必须包含一个名为 main、参数列表为空、返回类型为 int 的函数作为入口点。

### 检查阶段细化拆分
1. **全局元素类型检查**：
   * 确认 AST 的根节点（`CompUnit`）的子元素仅包含全局变量定义（`VarDecl`）、全局常量定义（`ConstDecl`）和全局函数定义（`FuncDef`）。
   * *（此项已由 AST 语法结构天然限制，但在语义阶段可作为结构不变性再次确认）*。
2. **`main` 入口函数存在性与签名校验**：
   * 在遍历全局声明的过程中，必须寻找名为 `main` 的函数。
   * 如果遍历结束未找到 `main` 符号，抛出语义错误。
   * 如果找到了 `main` 符号，必须检查其签名是否完全符合以下要求：
     * 函数的返回类型必须为 `"int"`。
     * 函数的形参列表（`params`）必须为空。
     * 否则，抛出语义错误。

## 2. 函数要求

### 原始文档要求

> ToyC 中的函数支持传入若干 int 类型的参数，返回值可以为 int 或 void 类型。函数只能声明在全局作用域中且名称唯一，不能声明在函数体内。函数不能作为值来储存、传递或运算。函数调用必须写在被调函数声明之后（支持函数内部调用自身）。返回类型为 int 的函数必须在每一条可能的执行路径上通过 return 语句返回一个 int 类型的值。返回类型为 void 的函数可以不包含 return 语句，如果包含，则不能有返回值。

### 检查阶段细化拆分

1. **类型范围校验**：
   * 函数形参类型必须为 `int`。
   * 函数返回类型只能为 `int` 或 `void`。

2. **全局作用域与唯一性校验**：
   * 函数定义（`FuncDef`）只允许出现在最外层（全局作用域），不允许嵌套定义。
   * 全局作用域下，函数名不能与已定义的全局变量、全局常量或其他函数名重名。

3. **函数“非值”特性校验（不允许将函数作为值传递/运算）**：
   * 在检查任意表达式节点时，如果是变量引用节点（`IdExpr`），需根据符号表解析其符号类型。
   * 如果解析到的符号类型为**函数**（`FuncDef`），且该节点**没有**作为函数调用节点（`CallExpr`）的目标函数名，则判定为非法。
   * 示例：`int a = main;`、`add + 1;` 必须报错。

4. **“声明后调用”与“递归支持”校验**：
   * **声明后调用**：当遇到函数调用（`CallExpr`）时，查找符号表。如果被调用的函数名尚未在当前符号表中注册，抛出“未定义函数”错误。
   * **递归支持**：函数可以在其自身的函数体内调用自己。
   * **扫描顺序设计**：
     * 遍历全局作用域下的每个元素。
     * 当遇到函数 `FuncDef f` 时：
       1. 首先校验 `f` 是否同名冲突。无冲突则立即将 `f` 的符号及签名注册到全局符号表中。
       2. 进入 `f` 的局部作用域，将 `f` 的形参注册到局部符号表中。
       3. 递归遍历 `f` 的函数体（此时因为 `f` 已经在全局表中，因此函数体内能正确解析 `f`，从而支持递归；但因为其后定义的函数尚未遍历和注册，函数体内无法调用后续定义的函数，从而强制了“声明后调用”）。

5. **返回值路径分析（针对 `int` 返回值函数）**：
   * 实现一个控制流分析辅助算法 `boolean checkAllPathsReturn(AST.Stmt stmt)`，判断特定语句（或语句块）是否能够**确保**在所有可能的执行分支上都执行了 `return <expr>;`。
   * **分支拆分规则**：
     * `ReturnStmt`：若包含表达式，返回 `true`；否则返回 `false`。
     * `BlockStmt`：顺序遍历其内部语句。若其中某个语句 `s` 满足 `checkAllPathsReturn(s) == true`，则该 Block 算作能确保返回。*(注：若 `s` 后面还有其他语句，说明存在不可达的死代码，可发出警告或报错)*。
     * `IfStmt`：**当且仅当**其 `thenStmt` 和 `elseStmt` 均非空且都满足 `checkAllPathsReturn(...) == true` 时，该 `IfStmt` 整体才算作能确保返回。如果缺失 `else` 分支，或任意分支不能确保返回，则返回 `false`。
     * `WhileStmt`：静态分析下 `while` 的条件可能在第一次就为假，循环体可能一次也不执行，故 `WhileStmt` 的返回值保障性为 `false`。
     * 其他语句（`AssignStmt`、`ExprStmt`、`BreakStmt`、`ContinueStmt`、`EmptyStmt`）：返回 `false`。
   * 校验：对于 `int` 函数，若 `checkAllPathsReturn(body)` 返回 `false`，编译报错。

6. **空返回值校验（针对 `void` 返回值函数）**：
   * 允许没有 `ReturnStmt` 语句。
   * 如果函数体内包含 `ReturnStmt`：
     * 该 `ReturnStmt` 的表达式必须为 `null`（即只能写 `return;`，不能写 `return 1;`）。
     * 否则，抛出编译语义错误。

## 3. 语句与控制流校验 (C-like Semantics & Loops)

### 原始文档要求
> ToyC 中的语句包括语句块、空语句、表达式语句、变量赋值、常量/变量声明、if-else 条件分支、while 循环、break、continue 和 return，它们的语法和语义与 C 语言一致：没有返回值的函数调用表达式不能作为 if/while 条件或赋值语句的右值；break 和 continue 只能出现在循环中等。

### 检查阶段细化拆分

1. **`break` 与 `continue` 循环嵌套校验**：
   * 在 `SemanticAnalyzer` 中维护一个整型状态 `loopDepth`。
   * 当进入 `WhileStmt` 节点时，`loopDepth++`；当离开 `WhileStmt` 节点时，`loopDepth--`。
   * 当遇到 `BreakStmt` 或 `ContinueStmt` 节点时，检查 `loopDepth` 的值：
     * 若 `loopDepth == 0`，则抛出语义错误，提示“break/continue 必须嵌套在循环中”。

2. **表达式值类型安全性校验（禁止使用 `void` 表达式）**：
   * 由于 ToyC 目前仅有 `int` 和 `void` 两种类型，且**唯有**调用返回值为 `void` 的函数（即 `CallExpr` 且其对应函数的返回类型为 `"void"`）才会产生 `void` 类型的表达式。其他所有的常数、标识符、一元/二元算术运算等表达式类型均为 `int`。
   * 因此，我们需要在 Visitor 中提供一个助手方法判定某个表达式的计算结果是否为 `void`：
     * 若 `expr` 是 `CallExpr`，且查找其函数定义发现返回类型为 `"void"`，则该表达式类型为 `void`。
     * 其余表达式类型一律为 `int`。
   * **类型安全性规则**：`void` 类型的表达式**不能**在任何需要 `int` 值的上下文中使用。在语义检查中，必须针对以下位置进行 `isVoid` 校验并报错：
     * **变量/常量初始化表达式**：`int a = f_void();` 判定非法。
     * **赋值语句右值**：`x = f_void();` 判定非法。
     * **条件控制表达式**：
       * `IfStmt` 中的 `cond` 表达式不能为 `void`。
       * `WhileStmt` 中的 `cond` 表达式不能为 `void`。
     * **二元运算符操作数**：二元运算（如 `+`、`-`、`*`、`/`、`==`、`&&` 等）的左、右操作数均不能为 `void`。
     * **一元运算符操作数**：一元运算（如 `+`、`-`、`!`）的操作数不能为 `void`。
     * **函数调用参数**：传给函数的实参表达式不能为 `void`。
     * **返回值表达式**：`int` 返回值的函数中，`return` 后的表达式不能为 `void`。

## 4. 变量与常量声明校验 (Variables & Constants)

### 原始文档要求
> ToyC 支持声明 32 位有符号整数（int）类型的局部变量、全局变量以及局部/全局常量（const）。
> - 初始化要求：所有声明（VarDecl 和 ConstDecl）必须带有初始化表达式。
> - 常量约束：由 const 关键字修饰的标识符为常量。常量的值在编译期必须能够确定（其初始化表达式只能包含数字字面量、已被声明的常量以及由它们组合而成的算术/逻辑运算）。常量一旦声明，其值不能被修改（不能作为赋值语句的左值）。
> - 作用域与生命周期：全局变量和全局常量在整个程序运行期间均有效。局部变量/常量的生命周期从声明点开始，到所在作用域结束时结束。函数形参在函数体内可见。语句块 Block 会创建新的嵌套作用域，内层作用域会屏蔽外层作用域的同名变量、常量或形参。变量和常量的使用必须发生在声明之后。

### 检查阶段细化拆分

1. **声明强制初始化校验**：
   * AST 层的 `VarDecl` 和 `ConstDecl` 语法已确保了必须包含初始表达式（`initExpr != null`），但在语义分析中我们可做一层防御性非空检查。

2. **编译期常量表达式校验与求值 (`evalConst`)**：
   * 对 `ConstDecl`（全局和局部常量）以及全局变量（`VarDecl` 且处于全局作用域），其 `initExpr` 必须是编译期常量表达式。
   * **求解与检测方法**：设计递归求值方法 `int evalConst(AST.Expr expr)`。
     * 若 `expr` 为 `NumberExpr`：返回其值。
     * 若 `expr` 为 `IdExpr`：去符号表查找对应名称。
       * 若该符号为已定义的**常量** (`ConstDecl`)，则返回其已求得的常量值。
       * 否则（属于普通变量或未定义），抛出语义错误：“非编译期常量”。
     * 若 `expr` 为 `UnaryExpr` 或 `BinaryExpr`：递归求出其子表达式的常量值，并按运算符计算结果。
       * **除零/模零校验**：若在求值过程中遇到除以 0 或模 0 运算，抛出语义错误：“常量表达式中除以 0 / 模 0”。
     * 其他表达式类型（如 `CallExpr`）：一律抛出语义错误：“非编译期常量”。
   * 对于 `ConstDecl`，求出的常量值需绑定在符号表对应的符号上，以便后续表达式求值中引用。

3. **常量只读（不能修改）校验**：
   * 遍历到 `AssignStmt`（例如 `name = expr;`）时：
     * 去符号表里解析 `name`。
     * 如果解析出的符号类型是**常量**（`isConst == true`），抛出语义错误：“无法为常量重新赋值”。

4. **作用域、嵌套与遮蔽（Shadowing）**：
   * 使用树状多级符号表结构（带 parent 指针的 `SymbolTable`）。
   * 每次进入语句块（`BlockStmt`）或函数（`FuncDef`），调用 `symTable.enterScope()` 开启一级新作用域。
   * 每次退出时，调用 `symTable.exitScope()`。
   * **重名/唯一性检查**：
     * 定义符号时，只在**当前这一层作用域**（`currentScope`）查找是否已存在同名符号。若有，则属于“重复定义”，抛出语义错误。
     * 如果外层作用域（`parentScope`）有同名符号，而当前层没有，则允许定义，并且由于符号表查询是自底向上（从 current 向 parent 递归），内层的符号会天然“遮蔽（Shadow）”外层的同名符号。
   * **生命周期与“声明后使用”**：
     * 符号表在当前 Block 退出时即被销毁，实现了局部变量的生命周期控制。
     * 由于我们对 AST 语句是顺序单次遍历，当变量 `x` 还没有被遍历（声明）到时，它不在符号表中，此时若在其前面有语句使用了 `x`，在符号表中自底向上都找不到 `x`，就会直接触发“未定义符号”报错，天然强制了“声明后使用”规则。

## 5. 表达式校验 (Expressions)

### 原始文档要求
> ToyC 支持逻辑运算、关系运算、算术运算、括号表达式、变量/常量引用、字面值和函数调用等 C 语言中的常见表达式。运算符的优先级、结合性和计算规则等与 C 语言一致：逻辑与、逻辑或运算符遵循“短路计算”规则；非零视为“真”、零视为“假”；除数不能为零等。

### 检查阶段细化拆分

1. **优先级、结合性与括号的语义继承**：
   * AST 树形结构在解析时已天然确定了运算符的执行顺序（叶子节点先执行）。因此在语义 `check` 阶段，我们只需递归遍历子表达式，无须做额外的优先级检查。

2. **布尔真假值概念（非零为真，零为假）**：
   * 在 C 语言规范下，任何 `int` 类型的值都能当作布尔值处理。由于我们前面进行了 `void` 类型拦截，只要表达式类型为 `int`，即可合法地用于 `IfStmt` / `WhileStmt` 的条件中，无需强制其必须为特定的布尔类。

3. **短路计算与静态语义检查的区分**：
   * **运行期行为**：逻辑与 `&&` 和逻辑或 `||` 必须在**执行期（生成代码时）**遵循短路求值规则（例如 `a && b`，如果 `a` 为假，不执行 `b`）。
   * **静态分析期（Check 阶段）行为**：在语义分析中，我们进行的是**全路径静态检查**。不管运行期是否短路，`&&` 和 `||` 左右两边的子表达式均必须通过语义检查（例如：两边都不能引用未定义的变量，不能包含类型错误）。

4. **静态“除以零 / 模零”检查**：
   * **编译期常量表达式中的除零**：
     * 在 `evalConst(Expr)` 对常量求值时，一旦检测到 `/ 0` 或 `% 0`，必须在编译期抛出语义错误。
   * **运行期普通除法**：
     * 对于变量参与的除法（如 `5 / x`），由生成的目标代码在运行期触发除零异常或遵循 RISC-V 规范（RISC-V 规定除以零不抛硬件异常，而是直接将目标寄存器置为 `-1`，模零置为被除数），编译器静态语义分析不强行报错。
